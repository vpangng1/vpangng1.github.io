Project Details

My Role: UI, Visual, and Graphic Designer

Duration: 4 Months

Tools: Figma, Adobe XD, Photoshop, Illustrator

Background

Fractal Link is a piece of software designed for people to do self-hosting using their own home computer and network. It is an alternative to third-party services and providers that host people's data like Amazon Web Services and Google Cloud. Fractal Link does this by leveraging existing self-hosting and virtualization technologies such as Docker. As the designer for this role, I was brought on to create visual assets and the UI for the website.

Research Insights

Using existing research that my team previously conducted, I was able to craft two different peronas which represent our model users. Our first model user, Chris, is someone who wants to do self-hosting but is unsure where to start. The other user, Greg, is someone who has experience self-hosting and wants to be able to share and self host easier.

Problem

The goal of Fractal Link is to provide an alternative to hosting that is affordable and accessible to everyone. How do we create an experience that caters to both new and advanced users that isn't confusing or frustrating?

Solution

We believe creating good navigation and flow for Fractal Link's product will create a pleasant and intuitive experience for new and advanced users. We will know this to be true when we see an increase in satisfaction and ratings.

Design Ideation & Sketches

Early concept sketches were done in Balsamiq for convenience. From looking at other competitor sites, we discovered that most of them had some demo or feature on the landing page that allowed users to get a feel of the product. We decided to base our design around this using a feature called the configurator that took in information from the user. This would generate a file where the user would then copy and paste the file into their own Docker file.

Initial Testing & Findings

Our team conducted testings on the team's staging website with two advanced users and two new users. I created a script to help guide me through the testing and recorded their responses using voice recording software on my mobile device. At the end of the test I also conducted a SUS Survey to calculate the usability score.


Findings: Both new users expressed confusion with regard to the configurator and had trouble figuring out where to find the information it was requesting for. Advanced users understood how to use the configurator but wanted to know what additional features were available. In addition, they stated it would be useful if they could get a live demo or preview.

Design Iteration & Visual Design

Our design team processed the feedback from our usability testings to develop design iterations to address user concerns. Our two goals for the next iteration were to rework the configurator and add a live demo feature.

Configurator

In the next iteration, the configurator was completely overhauled so that all users had to do was copy the code that was already on the screen and paste it in their docker file. When the visitor came to the site, the browser in the background would automatically generate a code inside the configurator using information from the visitor's computer. This code was different for every visitor that went to the site and eliminated the need for user input.

Live Demo

To address concerns for a live preview, our design team added demos of programs like Minecraft and WordPress on the landing page. This worked similarly to the configurator where the user copied the file to their own computer, but an instance of the program which was already configured would be available when the user launched the docker application.

2nd Round Testing & Findings

The second round of testing was done on the team's staging site based on the Adobe XD's sketches. Our team interviewed five users with different levels of docker experience. At the end of the test I also conducted a SUS Survey to calculate the usability score.


Findings: While three out of our five users had little difficulty with the configurator or demo, inexperienced users still found the overall process of copying and pasting the file on their computer somewhat difficult and confusing, especially if they have no docker experience.

Final Design Iterations

The goal for the next iteration was to simplify the process of installing application by eliminating steps for new users while preserving the traditional method for advanced users.

Install File

In the next iteration, the configurator was scrapped in favor of a packaged install file. When the user opened the download file, the program would install the app and docker if it wasn't already installed. Advanced users could still install apps using the command line interface and a page for new people who wanted to install apps the traditional way was provided.

App Catalog

Users had the option to download one of the apps from the landing page or to browse from the app catalog. This was not only done to make things more simpler for the user, but to create a sense of familiarity.

Final Round Testing & Findings

The third round of testing was done on the team's staging site based on the Adobe XD's sketches. Our team interviewed a total of five users. At the end of the test I also conducted a SUS Survey to calculate the usability score.


Findings: Users who were new or had little to no experience with docker greatly enjoyed the app catalog and thought installing docker/apps was intuitive. Experienced users preferred doing things through the command line interface but welcomed the alternative way to install apps.

Learnings

Understand your audience: One important lesson I learned was to not assume that everyone knows how to do something even when instructed. People have different levels of subject knowledge and experience. As a result some people need more guidance than others. In addition, it is necessary to be patient and take notes as much as possible.


Follow design standards: I've learned to stick to the design guidelines that have already been established in place in order to keep things consistent across the team. Working within the constraints also allowed me to focus on the objectives at hand and not stray too far from the project goals.
