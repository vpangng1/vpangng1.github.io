Project Details

My Role: UI Designer

Duration: 4 Months

Tools: Figma, Adobe XD, Photoshop, Illustrator

Background

Fractal Link is a computer application that allows users to safely publish websites, services, and applications from their personal computer. Fractal Link works with another application called Docker which "containerizes" applications to make them easier to create, deploy, and share with others. As the designer for the team, I was brought on to create visual assets and the UI for the website.

Problem

The goal of Fractal Link is to provide an alternative to hosting that is affordable and accessible to everyone. How do we create an experience that caters to both new and advanced users that isn't confusing or frustrating?

Solution

We believe creating good navigation and flow for Fractal Link's product will create a pleasant and intuitive experience for new and advanced users. We will know this to be true when we see an increase in satisfaction and ratings.

Research Insights

Our user research team discovered from interviews and internet sources that the target audience are users who have varying levels of self-hosting and Docker experience. In addition, based on survey results, our team identified four primary reasons users want to self-host.

Persona Development

Using existing research that my team previously conducted, our team was able to craft two different peronas which represent our model users. Our first model user, Chris, is someone who wants to try self-hosting but is unsure where to start. The other user, Greg, is an experienced user who wants to expand his self-hosting network.

User Journey

Because both our model users represent two types of people who would be using the product, our team created two different user journies. The user journey mapped out both user's anticipated behavior and experience as they engage with the product.

Design Ideation & Sketches

From looking at other competitor sites, we discovered that most of them had a call-to-action feature on the landing page that allowed users to get a feel of the product. We decided to base our design around this using a feature called the configurator that requested certain information from the user such as domain name, service name and port. This would generate a piece of code inside the configurator where the user would then copy and paste it into their own Docker file.

Initial Testing & Findings

Our team conducted testings on the team's staging website with two advanced users and two new users. I created a script to help guide me through the testing and recorded their responses using voice recording software on my mobile device. At the end of the test I also conducted a SUS Survey to calculate the usability score.



Findings: Both new users expressed confusion with regard to the configurator and had trouble figuring out where to find the information it was requesting for. Advanced users understood how to use the configurator but wanted to see a live demo of a working application.

Design Iteration & Visual Design

Our design team processed the feedback from our usability testings to develop design iterations to address user concerns. Our two goals for the next iteration were to rework the configurator and add a live demo feature.

Configurator

In the next iteration, the configurator was completely overhauled so that all users had to do was copy the code that was already on the screen and paste it in their Docker file. When the visitor came to the site, the configurator would automatically generate a code using information from the visitor's computer browser. This code was different for every visitor that went to the site and eliminated the need for user input.

Live Demo

To address concerns for a live preview, our design team added working demos of applications on the landing page. This worked similarly to the configurator where the user copied the code to their own Docker file, but an instance of the program which was already configured would be available when the user launched the Docker application.

2nd Round Testing & Findings

The second round of testing was done on the team's staging site based on the Adobe XD's sketches. Our team interviewed five users with different levels of Docker experience. At the end of the test I also conducted a SUS Survey to calculate the usability score.


Findings: While three out of our five users had little difficulty with using the configurator or demo, inexperienced users still found the overall process of copying and pasting the code into their Docker file somewhat difficult and confusing. Users also wished that there was clear cut instructions on how to use the configurator and install apps.

Final Design Iterations

The goal for the next iteration was to simplify the process of installing application by eliminating steps for new users while preserving the traditional method for advanced users.

Install File

In the next iteration, the configurator on the landing page was replaced with a packaged install file. When the user opened the file, the program would install the app as well as Docker if not already installed. Users could also visit the page for each respective app and either install them through the command line interface or configurator method.

Final Round Testing & Findings

The third round of testing was done on the team's staging site based on the Adobe XD's sketches. Our team interviewed a total of five users. At the end of the test I also conducted a SUS Survey to calculate the usability score.


Findings: Users who were new or had little to no experience with Docker greatly enjoyed installing apps through the install file method and thought the process was intuitive. Advanced users preferred doing things through the command line interface or configurator method but welcomed the alternative way to install apps.

Learnings

Understand your audience: One important lesson I learned was to not assume that everyone knows how to do something even when instructed. People have different levels of subject knowledge and experience. As a result some people need more guidance than others. In addition, it is necessary to be patient and take notes as much as possible.
